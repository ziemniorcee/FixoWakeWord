# Fikso wake-word detector: report draft

## 1. Motivation

The goal is a small Polish keyword-spotting system that detects "Fikso" in streaming microphone audio. The practical constraint is more important than leaderboard accuracy: training should finish in under one hour and the final model should run locally on a phone-class device.

## 2. Related course material

The main model uses Chapters 22-25: convolution over a time-frequency representation. Audio is converted to a 40-bin log-mel spectrogram, which exposes local frequency-time patterns such as syllables. Chapters 32-34 motivate a possible recurrent extension, but a CNN is sufficient for the current prototype and cheaper to train.

## 3. Data

The current training set is built from real recordings and augmented synthetic positives. All WAV files are mono 16-bit PCM at 16 kHz. `train.py` discovers files directly and uses a fixed stratified 70/15/15 split per source folder.

| Source folder | Label | Files | Role |
|---|---:|---:|---|
| `positive_real` | 1 | 812 | Real tablet/microphone recordings of "Fikso" |
| `positive_ai_augmented` | 1 | 405 | Synthetic standalone "Fikso" samples embedded in realistic windows |
| `negative_real` | 0 | 1416 | Real speech, room sound and background without the wake word |
| `hard_negative_real` | 0 | 69 | Similar-sounding words and false-trigger phrases without the wake word |

The augmented positive folder is generated by `augment_positive_ai.py` from standalone clips in `data/positive_ai`. Each generated sample places the word inside a 1.5-second window with 100-500 ms of silence or real negative background before and after it, a small random position offset, random gain, light reverb and background sampled from real negatives.

## 4. Model and streaming logic

The model receives 1.5-second audio windows. The audio frontend applies a speech-oriented filter before feature extraction: pre-emphasis with coefficient 0.97 and a mel filter range of 120-4800 Hz. This reduces low-frequency rumble and very high-frequency noise while keeping the main speech band.

After filtering, the model computes normalized 40-bin log-mel features and applies three convolutional blocks with 12, 24 and 32 channels. Adaptive pooling preserves a small time-frequency map before the binary classification head.

During streaming, windows advance every 250 ms. A lightweight RMS gate ignores silent frames. A wake event fires only after the calibrated number of consecutive scores exceeds the calibrated streaming threshold, followed by a cooldown to avoid duplicate detections. The Android export now synchronizes the Python frontend and Kotlin frontend, including pre-emphasis and the exported mel filter.

## 5. Training

The latest run used seed 42, Adam, binary cross-entropy, lightweight gain/noise augmentation and background mixing for positive samples. It trained on 2702 files total: 1892 train, 405 validation and 405 test files. The CPU run finished in 150.42 seconds and produced a 348 KB checkpoint.

## 6. Results

On the held-out test split, the latest checkpoint achieved:

| Metric | Value |
|---|---:|
| Threshold | 0.62 |
| Accuracy | 0.8864 |
| Precision | 0.8442 |
| Recall | 0.9180 |
| F1 | 0.8796 |

The calibrated streaming settings are:

| Setting | Value |
|---|---:|
| Streaming threshold | 0.87 |
| Required consecutive hits | 3 |
| Streaming recall on real positives | 0.7081 |
| Real negative audio | 72.753 min |
| False alarms | 217 |
| False alarms per hour | 178.962 |

The current system is much closer to a practical detector than the initial synthetic-only baseline, but the false-alarm rate is still the main limitation. The detector should be reported as a working prototype, not a production-ready wake-word system.

## 7. Error analysis

The remaining test errors in `results/metrics.json` show two important patterns. First, several hard negatives still score as positives, which means similar-sounding phrases need more coverage and possibly targeted hard-negative mining. Second, some real positive clips score below threshold, likely due to speaker variation, distance, noise or window placement.

The streaming calibration also shows that long-form real negative audio remains harder than isolated clip classification. The next useful evaluation is to listen to the false alarms from real negative sessions, group them by cause and add the most common failure modes to `hard_negative_real`.

## 8. Lessons learned

Offline clip accuracy is not enough for a wake-word system. Data realism, frontend consistency, streaming hysteresis, cooldown behavior and long-form background tests all matter. The most valuable improvements came from adding real recordings, generating realistic positive windows, filtering the speech band and keeping the Python and Android audio frontends synchronized.
